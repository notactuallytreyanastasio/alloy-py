### Changeset Pipeline

```python
from orm.src import (
    changeset, safe_identifier, TableDef, FieldDef,
    StringField, IntField, BoolField, SqlDefault, timestamps,
)

# Define schema
users_def = TableDef(
    safe_identifier("users"),
    [
        FieldDef(safe_identifier("name"), StringField(), False, None, False),
        FieldDef(safe_identifier("email"), StringField(), False, None, False),
        FieldDef(safe_identifier("age"), IntField(), True, None, False),
        FieldDef(safe_identifier("active"), BoolField(), False, None, False),
        *timestamps(),   # inserted_at + updated_at with DEFAULT
    ],
    safe_identifier("id"),   # custom primary key
)

# Build a changeset from user input
params = {"name": "Alice", "email": "alice@example.com", "age": "30"}
cs = (changeset(users_def, params)
      .cast([safe_identifier("name"), safe_identifier("email"), safe_identifier("age")])
      .validate_required([safe_identifier("name"), safe_identifier("email")])
      .validate_length(safe_identifier("name"), 1, 100)
      .validate_int(safe_identifier("age")))

# Check validity
if cs.is_valid:
    insert_sql = cs.to_insert_sql().to_string()
    # => INSERT INTO users (name, email, age) VALUES ('Alice', 'alice@example.com', 30)

# Update existing record
update_sql = cs.to_update_sql(SqlInt32(42)).to_string()
# => UPDATE users SET name = 'Alice', email = 'alice@example.com', age = 30 WHERE id = 42

# Additional validations
(cs.validate_int64(safe_identifier("big_num"))
   .validate_float(safe_identifier("score"))
   .validate_bool(safe_identifier("active"))
   .validate_inclusion(safe_identifier("role"), ["admin", "user", "guest"])
   .validate_exclusion(safe_identifier("status"), ["banned"])
   .validate_number(safe_identifier("age"), 0, 150)
   .validate_acceptance(safe_identifier("terms"))
   .validate_confirmation(safe_identifier("password"), safe_identifier("password_confirmation"))
   .validate_contains(safe_identifier("bio"), "keyword")
   .validate_starts_with(safe_identifier("code"), "PRE-")
   .validate_ends_with(safe_identifier("email"), ".com"))

# Data manipulation
(cs.put_change(safe_identifier("updated_at"), "2026-03-13")
   .get_change(safe_identifier("name"))         # => "Alice"
   .delete_change(safe_identifier("age")))      # remove a field

# Access errors
cs.errors  # List of validation error strings
```

## Security Model

The Alloy ORM is built on a defense-in-depth approach to SQL injection prevention, using Temper's type system to enforce safety at compile time. The compiled Python output preserves all safety invariants.

### Defense Layers

**Layer 1: SafeIdentifier (CWE-89 -- table/column names)**

Table and column names must pass through `safe_identifier()`, which validates against `[a-zA-Z_][a-zA-Z0-9_]*`. The internal `ValidatedIdentifier` class is not exported -- external code cannot construct one without validation. This is the only path to `append_safe` at runtime, closing off identifier-based injection entirely.

```python
safe_identifier("users")        # => ValidatedIdentifier (OK)
safe_identifier("users; DROP")  # => bubble (rejected)
safe_identifier("table--name")  # => bubble (rejected)
```

**Layer 2: Typed SqlPart hierarchy (CWE-89 -- value escaping)**

User values never touch `append_safe`. Each type gets its own `SqlPart` subclass with type-appropriate rendering:

| Type | Rendering | Example |
|------|-----------|---------|
| `SqlString` | Quote-wraps with `'` escaping (`'` becomes `''`) | `'O''Brien'` |
| `SqlInt32` / `SqlInt64` | Bare integer via `str()` | `42` |
| `SqlBoolean` | Literal `TRUE` / `FALSE` | `TRUE` |
| `SqlFloat64` | Bare float (NaN/Infinity render as `NULL`) | `3.14` |
| `SqlDate` | Quote-wrapped date string with `'` escaping | `'2026-03-13'` |
| `SqlDefault` | Literal `DEFAULT` keyword | `DEFAULT` |

The `SqlPart` interface is sealed -- the compiler enforces exhaustive handling.

**Layer 3: SqlBuilder separation (CWE-89 -- structural safety)**

`SqlBuilder` enforces a clean separation between SQL structure and data:
- `append_safe(str)` -- for SQL keywords and validated identifiers only
- `append_string(val)` -- wraps in `SqlString` (escaped)
- `append_int32(val)` -- wraps in `SqlInt32` (bare integer)

There is no `append_raw` or bypass method.

**Layer 4: Changeset pipeline (CWE-915 -- mass assignment)**

`Changeset` is a sealed interface -- `ChangesetImpl` is not exported, so construction only happens through the `changeset()` factory. The pipeline enforces:
- `cast(allowed_fields)` -- field whitelisting via `List[SafeIdentifier]`, prevents mass assignment
- `validate_required(fields)` -- non-nullable enforcement
- `to_insert_sql()` / `to_update_sql(id)` -- independently enforces non-nullable fields
- Raw `params` are never exposed -- only whitelisted `changes` are readable

**Layer 5: Query builder (CWE-89, CWE-400)**

`Query` requires `SafeIdentifier` for table names, column selections, ORDER BY clauses, and JOIN table names. WHERE and ON conditions accept only `SqlFragment` (never raw strings). `safe_to_sql(default_limit)` enforces result set bounds (CWE-400). `UpdateQuery` and `DeleteQuery` bubble on empty WHERE clauses, preventing accidental full-table mutations.

### Per-Phase Security Analysis

**JOIN Feature Security** -- JOIN keywords use a sealed `JoinType` interface with hardcoded strings. Table names require `SafeIdentifier`. ON conditions accept only `SqlFragment`. The `col()` helper produces qualified column references from two `SafeIdentifier` values joined by a hardcoded `.` separator.

**WHERE Clause Enrichment** -- `WhereClause` is sealed (`AndCondition`/`OrCondition`). Convenience methods (`where_null`, `where_in`, `where_between`, `where_like`, `where_ilike`) build fragments internally using only validated identifiers and typed values. Empty IN list produces `1 = 0`. LIKE/ILIKE patterns go through `SqlString` escaping.

**Aggregation** -- All 6 aggregate functions use hardcoded function names via `append_safe`. Field names require `SafeIdentifier`. `select_expr`, `group_by`, `having`/`or_having` reuse existing safe types. `distinct()` selects between two hardcoded strings.

**Set Operations and Subqueries** -- Set operation keywords (`UNION`, `INTERSECT`, `EXCEPT`) are hardcoded literals. Both operands are `Query` objects. Sub-SELECTs are parenthesized. `subquery()` alias requires `SafeIdentifier`. `exists_sql()` wraps in hardcoded `EXISTS (...)`. `where_in_subquery()` combines `SafeIdentifier` field with `Query` subquery.

**Batch UPDATE/DELETE** -- `SetClause.field` requires `SafeIdentifier`. `SetClause.value` uses `SqlPart` (sealed, type-dispatched). `UpdateQuery` and `DeleteQuery` bubble on empty WHERE clauses, preventing accidental full-table mutations.

**Extended Query Features** -- `NullsPosition` and `CrossJoin` use sealed interfaces with hardcoded keywords. `LockMode` (`ForUpdate`/`ForShare`) uses sealed interfaces with hardcoded strings. Nullable `on_condition` omits the ON clause entirely when null.

**Changeset Enrichment** -- All validation methods operate on the `changes` dict (string validation only, no SQL generation). Field parameters require `SafeIdentifier`. Values only reach SQL through `to_insert_sql()`/`to_update_sql()` which use type-dispatched escaping. Immutable pattern preserved.

**Schema Enrichment** -- `TableDef.primary_key` uses `SafeIdentifier`. `pk_name()` returns `SafeIdentifier.sql_value` or hardcoded `"id"`. `FieldDef.default_value` uses `SqlPart` (sealed). `SqlDefault` renders hardcoded `"DEFAULT"`. Virtual fields are excluded from SQL entirely.

## Getting Started

### Installation

Copy (or vendor) the `orm/`, `std/`, and `temper-core/` directories into your project. The Temper runtime dependency `temper_core` is included in the `temper-core/` directory.

```
your-project/
  vendor/
    orm/          <-- from this repo
    std/          <-- from this repo
    temper-core/  <-- from this repo
  app.py
```

Ensure the vendor directory is on your Python path, or use relative imports.

### Minimal Example

```python
import sys
sys.path.insert(0, "vendor")

from orm.src import from_, safe_identifier, changeset, TableDef, FieldDef, StringField, IntField, SqlInt32

# Define a table
todos_table = TableDef(
    safe_identifier("todos"),
    [
        FieldDef(safe_identifier("title"), StringField(), False, None, False),
        FieldDef(safe_identifier("completed"), IntField(), False, None, False),
    ],
    None,
)

# SELECT query
select_sql = (from_(safe_identifier("todos"))
    .order_by(safe_identifier("created_at"), True)
    .limit(50)
    .to_sql()
    .to_string())
print(select_sql)
# => SELECT * FROM todos ORDER BY created_at ASC LIMIT 50

# INSERT via changeset
params = {"title": "Buy milk", "completed": "0"}
cs = (changeset(todos_table, params)
    .cast([safe_identifier("title"), safe_identifier("completed")])
    .validate_required([safe_identifier("title")]))

if cs.is_valid:
    print(cs.to_insert_sql().to_string())
    # => INSERT INTO todos (title, completed) VALUES ('Buy milk', 0)
```

## Architecture

This repository is updated automatically by CI. The cascade works as follows:

```
alloy (Temper source)
  |
  | temper build -b py
  v
temper.out/py/
  |
  | CI publish (push to this repo)
  v
alloy-py (this repo)
  |
  | notify-app.yml workflow (triggered on push)
  v
alloy-py-app (demo application)
  |
  | vendors orm/, std/, temper-core/ into vendor/
  v
Application uses the ORM
```

A push to `main` in the [Alloy source repository](https://github.com/notactuallytreyanastasio/alloy) triggers CI, which builds the Temper compiler, compiles the ORM for all 6 backends, runs the test suite, and pushes the compiled Python output to this repository. The `notify-app.yml` workflow in this repo then automatically vendors the updated ORM into the [alloy-py-app](https://github.com/notactuallytreyanastasio/alloy-py-app) demo application.
